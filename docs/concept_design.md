# 경량 GPS 비의존 스프린트 드론 설계 아이디어

## 1. 핵심 컨셉

### 제안 이름

**Ingenuity-Inspired Lightweight GPS-Denied Sprint Drone**

또는

**Lightweight GPS-Denied Perception-Aware Sprint Navigation System**

### 한 줄 요약

화성 탐사 헬리콥터 Ingenuity의 GPS 비의존 항법 개념을 참고하여, GPS 없이 하향 카메라와 IMU로 격자 환경을 인식하고, 경량화된 기체의 빠른 가속 및 감속 성능을 활용해 미션 시간을 최소화하는 자율비행 시스템을 개발한다.

### 핵심 차별점

- GPS를 사용하지 않는 비전 및 관성 기반 항법
- 경기장의 격자선을 시각적 항법 기준으로 활용
- 규정 고도 2m를 안정적으로 유지하는 저고도 비행
- 직선 구간에서 가속도를 적극 활용하는 스프린트 비행
- 마커 인식 구간에서는 흔들림을 줄이는 인식 친화 제어
- 짐벌, 고성능 보드, 대용량 배터리 의존도를 낮춘 경량 기체
- 무거운 하드웨어 대신 소프트웨어 기반 안정화 및 경로 최적화로 성능 확보

---

## 2. 전체 개발 철학

일반적인 자율비행 시스템은 GPS, 무거운 센서, 짐벌, 고성능 컴퓨팅 보드에 의존해 안정성과 인식률을 확보하는 경우가 많다. 본 시스템은 반대로 기체를 최대한 가볍게 구성하고, 경기장에 이미 존재하는 격자 구조와 하향 카메라, IMU를 적극 활용해 빠르고 가벼운 자율비행을 구현하는 것을 목표로 한다.

즉, 본 시스템의 방향은 다음과 같다.

- 하드웨어는 가볍게
- 센서는 최소한으로
- 항법은 GPS 없이
- 직선 구간은 빠르게
- 인식 순간은 안정적으로
- 부족한 안정성은 제어와 비전 알고리즘으로 보완

---

## 3. Ingenuity 아이디어 차용 방향

화성 탐사 헬리콥터 Ingenuity는 GPS를 사용할 수 없는 환경에서 하향 카메라와 관성 센서를 이용해 비행했다. 이 철학을 대회 환경에 맞게 재구성한다.

### 적용 방식

- GPS 대신 하향 카메라와 IMU 기반 위치 추정 사용
- 지면의 시각적 특징 대신 경기장 격자선을 주요 특징으로 사용
- 격자 교차점, 선분, 꼭짓점, ArUco 마커를 위치 보정 기준으로 활용
- IMU의 짧은 시간 응답성과 카메라의 누적 위치 보정 능력을 결합

### 문서용 표현

> 본 팀은 화성 탐사 헬리콥터 Ingenuity의 GPS 비의존 항법 철학을 참고하여, GPS에 의존하지 않고 하향 카메라와 IMU를 기반으로 격자 환경을 인식하는 경량 자율비행 시스템을 개발한다.

---

## 4. GPS 비의존 항법 구조

### 센서 구성

- 하향 고정 카메라
- IMU
- 기압계 또는 거리 센서 기반 고도 추정
- Optical Flow 또는 영상 특징 추적
- ArUco 마커 인식

### 위치 추정 개념

1. IMU로 짧은 시간의 자세 변화, 가속도, 각속도 추정
2. 하향 카메라로 격자선과 교차점 인식
3. Optical Flow로 지면 기준 이동량 추정
4. 격자 좌표계에 현재 위치를 보정
5. ArUco 마커가 보이면 절대적 랜드마크처럼 활용

### 격자 기반 보정

경기장의 격자는 3m 간격으로 구성되어 있으므로, 격자 교차점을 기준 좌표로 정의할 수 있다. 드론은 카메라로 검출한 격자선과 교차점을 이용해 자신의 위치를 격자 좌표계에 정렬한다.

예시:

```text
Grid Origin: 버티포트 또는 미션 영역 기준점
Grid Unit: 3m
Node: 격자 교차점 또는 꼭짓점
Edge: 격자선 구간
Marker: 특정 Node 또는 Vertex에 연결된 랜드마크
```

## 5. Grid-SLAM 또는 Grid-Based Localization

### 아이디어

미션장의 격자를 단순히 따라가야 하는 선이 아니라, 드론이 자신의 위치를 추정하기 위한 시각적 지도 구조로 사용한다.

### 주요 기능

- 격자선 검출
- 격자 교차점 검출
- 현재 비행 방향과 격자 방향 정렬
- 격자 좌표계 기준 위치 보정
- ArUco 마커 발견 위치 저장
- 구조 순회 단계에서 저장된 마커 좌표 재방문

### 장점

- GPS 오차 또는 GPS 미사용 상황에서도 위치 추정 가능
- 미션장 구조를 소프트웨어적으로 활용하므로 자체 개발 요소로 어필 가능
- 격자선 이탈 감지 및 복귀 로직과 자연스럽게 연결됨

### 문서용 표현

> 본 팀은 경기장의 격자 구조를 단순 비행 경로가 아닌 시각적 항법 기준점으로 활용하는 Grid-Based Localization 기술을 개발한다.

## 5-1. 저고도 인식 최적화 비행

### 핵심 방향

하향 카메라 기반 격자선 인식과 ArUco 마커 인식은 고도가 낮을수록 유리하다. 본 대회 규정 고도가 2m이므로, 시스템은 2m 저고도에서 안정적인 인식과 빠른 스프린트 비행을 동시에 달성하도록 설계한다.

### 기준 고도

```text
대회 기준 고도: 2m
개발 및 테스트 기준 고도: 2m
제어 목표: 2m 부근 안정 유지
```

따라서 고도 제어, 카메라 화각, 마커 인식 거리, Optical Flow scale, 스프린트 속도 제한은 모두 2m 기준으로 맞춘다.

### 2m 고도의 장점

- ArUco 마커가 영상에서 더 크게 보임
- 격자선과 교차점 검출이 쉬워짐
- Optical Flow 기반 이동량 추정 정확도 향상
- 버티포트 중심 검출 및 착륙 보정에 유리
- 동일 카메라 해상도에서 인식 가능한 픽셀 정보 증가
- 고도 변화에 따른 위치 추정 오차를 줄이기 쉬움

### 2m 고도의 주의점

저고도 비행은 인식에는 유리하지만, 고속 스프린트와 결합하면 다음 문제가 생길 수 있다.

- 화면에서 지면이 빠르게 흘러 Optical Flow가 커짐
- 카메라 FPS가 낮으면 motion blur가 증가
- 제동 거리가 부족하면 격자 교차점이나 마커를 지나칠 수 있음
- 작은 자세 흔들림도 영상 내 위치 변화로 크게 나타남
- 지면 효과와 프로펠러 바람이 비행 안정성에 영향을 줄 수 있음

따라서 저고도에서는 속도 프로파일을 고도와 함께 제한한다.

### 고도 연동 속도 제한

고도가 낮을수록 인식 안정성을 위해 최대 속도와 가속도를 낮춘다.

```text
Altitude Mode: Low Perception Altitude
Target Altitude: 2m
Control Policy:
  - 최고 속도 제한
  - 최대 가속도 제한
  - 제동 거리 여유 증가
  - 카메라 FPS 기준 motion blur 제한
  - 마커 후보 접근 시 조기 감속
```

### 스프린트 플래너와의 연계

Minimum-Time Sprint Search Planner는 고도를 고정된 배경 조건으로 보지 않고, 시간 비용 계산에 포함한다.

예시:

```text
낮은 고도:
  인식 정확도 증가
  허용 속도 감소
  조기 감속 필요

높은 고도:
  인식 정확도 감소
  시야 범위 증가
  허용 속도 증가 가능
```

따라서 2m 고도에서 실제로 가장 빠른 비행이 가능한 속도 상한을 실험으로 확인하고, 해당 값을 비용 함수에 반영한다.

### 문서용 표현

> 본 팀은 대회 규정 고도인 2m를 기준으로 하향 카메라 기반 격자선 및 ArUco 인식 알고리즘을 설계한다. 2m 저고도에서는 마커와 격자선이 영상에서 크게 관측되어 인식에 유리하지만, 고속 비행 시 motion blur와 제동 거리 부족 문제가 발생할 수 있다. 이를 보완하기 위해 2m 고도 기준 속도 상한, 조기 감속, 흔들림 억제 제어를 함께 적용한다.

## 6. 가속도 기반 스프린트 비행

### 핵심 아이디어

드론이 직선으로 이동할 때는 가속도와 감속도를 적극 활용해 가능한 빠르게 이동한다. 단순히 고정 속도로 격자선을 따라가는 것이 아니라, 각 직선 구간의 길이와 다음 행동을 고려해 속도 프로파일을 생성한다.

### 속도 프로파일

구간별로 다음과 같은 형태를 적용한다.

```text
가속 -> 순항 -> 감속
```

짧은 구간에서는 순항 없이 다음과 같이 동작할 수 있다.

```text
가속 -> 감속
```

### 제동 거리 계산

현재 속도에서 멈추기 위해 필요한 거리를 계산한다.

```text
d_stop = v^2 / (2 * a_brake)
```

- `d_stop`: 정지 필요 거리
- `v`: 현재 속도
- `a_brake`: 허용 가능한 감속도

### 제어 판단

```text
남은 거리 > 정지 필요 거리 + 안전 여유
```

이면 계속 가속하거나 순항하고, 그렇지 않으면 감속을 시작한다.

### 적용 예시

- 긴 직선 격자 구간: 적극 가속
- 교차점 접근: 감속
- 회전 지점 접근: 미리 감속
- 마커 후보 발견: 즉시 감속 및 인식 안정 모드 진입
- 미션 경계 접근: 속도 제한

### 문서용 표현

> 일반적인 자율비행 방식은 고정 속도로 격자선을 추종하지만, 본 팀은 미션장의 격자 구조를 주행 가능한 시각 회랑으로 해석하고 각 직선 회랑의 길이에 따라 가속, 순항, 감속 속도 프로파일을 동적으로 생성한다.

## 7. Minimum-Time Sprint Search Planning

### 개념

탐색 알고리즘은 단순 DFS, BFS, 최단거리 탐색처럼 그래프 상의 이동 횟수나 거리만 줄이는 방식이 아니라, 실제 드론이 물리적으로 가장 빠르게 미션을 수행할 수 있는 경로를 선택하는 것을 목표로 한다.

격자선을 하나의 비행 회랑으로 보고, 각 후보 경로에 대해 가속, 감속, 회전, 정지, 호버링, 인식 안정화에 필요한 시간을 계산한다. 그 결과 단순히 거리가 짧은 경로가 아니라, 실제 예상 비행 시간이 가장 짧은 경로를 선택한다.

### 핵심 차이

일반적인 그래프 탐색:

```text
비용 = 이동 거리 또는 방문 노드 수
```

본 시스템의 탐색:

```text
비용 = 실제 예상 수행 시간
```

따라서 다음과 같은 상황에서는 거리가 조금 길어도 더 빠른 경로를 선택할 수 있다.

- 짧지만 회전이 많은 경로
- 길지만 직선 스프린트가 가능한 경로
- 마커 후보를 안정적으로 확인하기 쉬운 경로
- 감속과 재가속 횟수가 적은 경로

### 물리 기반 시간 비용 모델

각 후보 경로의 비용은 다음 항목을 합산한다.

```text
T_total =
  T_accel_decel
  + T_cruise
  + T_turn
  + T_marker_check
  + T_hover
  + T_stabilize
  + T_safety_margin
```

항목 설명:

| 항목 | 의미 |
|------|------|
| `T_accel_decel` | 가속 및 감속에 필요한 시간 |
| `T_cruise` | 최고 속도 또는 제한 속도로 이동하는 시간 |
| `T_turn` | 회전 또는 방향 전환으로 인한 시간 손실 |
| `T_marker_check` | 마커 후보를 확인하기 위한 감속 및 관측 시간 |
| `T_hover` | 확정 마커에서 3초 호버링하는 시간 |
| `T_stabilize` | 흔들림을 줄이기 위한 안정화 시간 |
| `T_safety_margin` | 경계, 교차점, 인식 실패 가능성에 대한 여유 |

### 직선 구간 시간 계산

직선 회랑의 길이 `L`, 최대 속도 `v_max`, 최대 가속도 `a_max`, 최대 감속도 `a_brake`가 주어졌을 때, 해당 구간을 몇 초에 통과할 수 있는지 계산한다.

가속 후 감속만 가능한 짧은 구간:

```text
L < d_accel + d_brake
```

이면 최고 속도에 도달하지 못하므로 삼각형 속도 프로파일을 사용한다.

```text
가속 -> 감속
```

충분히 긴 구간:

```text
L >= d_accel + d_brake
```

이면 사다리꼴 속도 프로파일을 사용한다.

```text
가속 -> 순항 -> 감속
```

### 회전 비용 반영

격자 탐색에서 빠른 경로를 만들려면 회전 횟수를 줄이는 것이 매우 중요하다. 회전은 단순히 yaw만 바꾸는 문제가 아니라, 수평 속도 감소, 자세 안정화, 카메라 영상 흔들림 증가를 함께 유발한다.

회전 비용은 다음과 같이 반영한다.

```text
T_turn = f(회전각, 현재 속도, 허용 yaw rate, 안정화 시간)
```

예시:

| 회전 형태 | 비용 |
|----------|------|
| 직진 유지 | 매우 낮음 |
| 90도 회전 | 중간 |
| 180도 유턴 | 높음 |
| 고속 진입 후 급회전 | 매우 높음 |

따라서 탐색기는 가능한 한 긴 직선 주행을 유지하고, 불필요한 지그재그 경로를 피한다.

### 경로 계획 방식

- 격자 교차점과 꼭짓점을 노드로 정의
- 격자선을 엣지로 정의
- 긴 직선 구간은 하나의 스프린트 회랑으로 묶음
- 회전 지점과 마커 후보 지점은 감속 포인트로 지정
- 각 후보 경로에 대해 물리 기반 예상 시간을 계산
- 단순 거리 최단이 아니라 예상 시간 최소 경로를 선택
- 이미 탐색한 구간이라도 고속 직선 연결에 유리하면 재사용 가능
- 미탐색 구간을 커버하면서도 감속/회전 횟수가 적은 경로를 우선 선택

### 탐색 후보 생성 방식

탐색기는 매 순간 하나의 다음 노드만 고르는 것이 아니라, 여러 개의 후보 스프린트 회랑을 생성한 뒤 예상 시간을 비교한다.

후보 예시:

```text
Candidate A: 가까운 교차점까지 이동 후 회전
Candidate B: 같은 방향으로 2개 격자 더 직진 후 회전
Candidate C: 이미 탐색한 선을 재사용해 긴 직선으로 이동
Candidate D: 마커 후보 밀도가 높은 영역으로 빠르게 이동
```

각 후보는 다음 기준으로 평가한다.

- 예상 이동 시간
- 회전 횟수
- 감속 및 재가속 횟수
- 미탐색 구간 커버율
- 마커 후보 발견 가능성
- 경계 접근 위험
- 인식 안정성

### 시간 최적화 탐색 비용 함수

탐색 비용 함수는 다음과 같이 정의할 수 있다.

```text
Cost(route) =
  w_time * T_total
  + w_turn * N_turn
  + w_stop * N_stop
  + w_altitude * AltitudePenalty
  + w_risk * Risk
  - w_coverage * CoverageGain
  - w_marker * MarkerLikelihood
```

의미:

| 항목 | 의미 |
|------|------|
| `T_total` | 물리 기반 예상 수행 시간 |
| `N_turn` | 회전 횟수 |
| `N_stop` | 정지 또는 강한 감속 횟수 |
| `AltitudePenalty` | 고도에 따른 인식 안정성 및 속도 제한 비용 |
| `Risk` | 경계 접근, 흔들림, 인식 실패 위험 |
| `CoverageGain` | 새롭게 탐색되는 격자 구간 |
| `MarkerLikelihood` | 마커가 있을 가능성이 높은 후보 위치 |

최종적으로는 `Cost(route)`가 가장 낮은 후보를 선택한다.

### 온라인 재계획

드론은 실제 비행 중 다음 정보를 계속 업데이트한다.

- 현재 속도
- 현재 가속도
- 현재 자세 안정도
- 남은 제동 거리
- 격자선 인식 상태
- 마커 후보 위치
- 미탐색 구간

따라서 사전에 정해둔 전체 경로를 그대로 따르는 것이 아니라, 매 교차점 또는 일정 주기마다 현재 동역학 상태를 반영해 경로를 다시 계산한다.

```text
현재 상태 측정
-> 후보 스프린트 회랑 생성
-> 물리 기반 시간 비용 계산
-> 최소 시간 후보 선택
-> 속도 프로파일 생성
-> 비행 중 마커 후보 발견 시 즉시 재계획
```

### 예시 판단

단순 거리 기준으로는 A가 짧을 수 있다.

```text
A 경로: 9m 이동, 90도 회전 3회
B 경로: 12m 이동, 90도 회전 1회
```

하지만 실제 드론은 회전할 때 감속하고 다시 가속해야 하므로, B 경로가 더 빠를 수 있다.

```text
A 경로 예상 시간 = 짧은 이동 + 회전 손실 3회
B 경로 예상 시간 = 긴 직선 스프린트 + 회전 손실 1회
```

본 시스템은 이 경우 B 경로를 선택한다.

### 모드 구분

```text
Grid Search Mode
Sprint Mode
Marker Candidate Mode
Hover Confirm Mode
Rescue Route Mode
Return Mode
Landing Mode
```

### 장점

- 실제 기체 동역학을 반영한 시간 단축
- 긴 직선 구간에서 스프린트 성능 극대화
- 불필요한 회전, 감속, 재가속 감소
- 회전 및 인식 구간에서 안정성 확보
- 단순 DFS/BFS보다 대회 평가 기준인 임무 수행 시간에 직접 최적화
- 미션 수행 시간 평가 항목에 직접적으로 기여

### 문서용 표현

> 본 팀의 탐색 알고리즘은 단순 DFS 또는 최단거리 기반 그래프 탐색이 아니라, 드론의 가속도, 감속도, 최고 속도, 회전 손실, 인식 안정화 시간을 반영한 Minimum-Time Sprint Search Planner이다. 이를 통해 거리가 조금 길더라도 직선 가속 비행이 가능한 경로를 우선 선택하여 실제 임무 수행 시간을 최소화한다.

## 8. ArUco Marker Confidence Filter

### 문제의식

고속 비행 중에는 마커가 한두 프레임만 보이거나, 흔들림 때문에 오인식이 발생할 수 있다. 따라서 한 프레임에서 마커가 보였다고 바로 발견 처리하지 않고, 여러 프레임의 관측 결과를 누적해 신뢰도를 계산한다.

### 주요 기준

- 동일 ID가 여러 프레임에서 반복 검출되는지
- 마커의 화면상 크기가 안정적인지
- 마커의 회전각 변화가 급격하지 않은지
- 추정 위치가 이전 프레임과 일관적인지
- IMU 기준 기체 흔들림이 과도하지 않은 프레임인지
- Optical Flow가 비정상적으로 큰 프레임은 아닌지

### 상태 구분

```text
Not Detected
Candidate
Tracking
Confirmed
Visited
```

### 동작 흐름

1. 마커 후보 검출
2. 드론 감속
3. 인식 안정 모드 진입
4. 여러 프레임의 ID 및 위치 누적
5. 신뢰도가 기준 이상이면 발견 확정
6. 해당 위치에서 3초 호버링
7. 마커 위치와 ID 저장

### 문서용 표현

> 단일 프레임 기반 인식의 취약점을 보완하기 위해 누적 관측 기반 Marker Confidence Filter를 자체 개발한다.

## 9. Perception-Aware Anti-Sway Flight Control

### 핵심 아이디어

드론이 빠르게 가속하거나 방향을 바꾸면 좌우 흔들림, roll/pitch 변화, yaw rate 증가가 발생할 수 있다. 이는 하향 카메라 영상의 흔들림으로 이어져 ArUco 인식률을 떨어뜨린다. 따라서 비행 제어 자체가 인식 성능을 고려하도록 설계한다.

### 목표

- 고속 비행과 안정적 인식의 균형 확보
- 마커 후보 발견 시 흔들림을 빠르게 억제
- 기체 자세가 불안정한 프레임은 인식 확정에서 제외
- 짐벌 없이도 하향 고정 카메라 인식률 유지

### 흔들림 감지 지표

- roll 각도
- pitch 각도
- yaw rate
- lateral acceleration
- jerk
- Optical Flow 변화량
- 영상 내 격자선 기울기 변화
- ArUco confidence 급락

### 인식 안정 모드

```text
Sprint Mode
-> Marker Candidate Detected
-> Anti-Sway Stabilization Mode
-> Hover Confirm
-> Sprint Mode
```

### 안정 모드에서 제한할 항목

- 최대 수평 속도
- 최대 수평 가속도
- 최대 jerk
- 최대 roll/pitch 각도
- 최대 yaw rate
- 급격한 방향 전환

### 문서용 표현

> 본 팀은 고속 직선 비행 중 발생하는 좌우 흔들림과 기체 자세 변화가 하향 카메라 기반 ArUco 인식 성능을 저하시킬 수 있다는 점에 착안하여, 비전 인식 안정성을 고려한 Perception-Aware 비행 제어 계층을 개발한다.

## 10. PX4 수정 및 제어 확장 방향

### 단계적 접근

PX4 소스코드를 바로 수정하는 것은 리스크가 크므로, 개발 단계에 따라 접근한다.

### 1단계: ROS2 Offboard Setpoint Shaping

PX4 자체 코드는 수정하지 않고, ROS2 미션 매니저에서 속도, 가속도, yaw setpoint를 안정적으로 생성해 전달한다.

적용 항목:

- 스프린트 구간 속도 프로파일 생성
- 마커 후보 발견 시 감속 setpoint 전달
- 호버링 setpoint 유지
- 격자선 이탈 시 복귀 setpoint 생성

### 2단계: PX4 파라미터 튜닝

PX4의 수평 속도, 가속도, jerk, tilt 제한과 관련된 파라미터를 조정해 기체 흔들림을 줄인다.

검토 항목:

- 최대 수평 속도 제한
- 최대 수평 가속도 제한
- jerk 제한
- tilt angle 제한
- position controller gain
- velocity controller gain

### 3단계: PX4 FlightTask 또는 Position Controller 수정 검토

필요 시 PX4 소스코드 레벨에서 비전 인식 상태를 반영하는 제어 로직을 추가한다.

가능한 수정 아이디어:

- `perception_mode` 상태 추가
- 마커 후보 발견 시 동적 가속도 제한 적용
- 인식 안정 모드에서 lateral acceleration 제한 강화
- roll/pitch 응답을 더 부드럽게 만드는 jerk limiter 추가
- 흔들림 지표가 높을 때 자동 감속
- 비전 confidence가 낮을 때 hover 안정화 우선

### 문서용 표현

> 기본 구현은 ROS2 Offboard 제어를 통해 PX4에 속도 및 가속도 제한이 반영된 setpoint를 전달하고, 필요 시 PX4 FlightTask 또는 Position Controller 계층을 수정하여 비전 인식 상태에 따라 jerk, tilt angle, lateral acceleration 제한값을 동적으로 조정한다.

## 11. 짐벌 없는 영상 안정화

### 하드웨어 방향

짐벌을 달지 않고 고정식 하향 카메라를 사용한다. 짐벌 미탑재로 생기는 영상 흔들림은 제어와 소프트웨어 보정으로 해결한다.

### 보정 방식

- IMU 각속도 기반 흔들림 추정
- 카메라 timestamp와 IMU timestamp 동기화
- Optical Flow 기반 프레임 간 이동량 추정
- 격자선 방향을 이용한 영상 회전 보정
- 자세 변화가 큰 프레임은 신뢰도 낮게 반영
- 마커 후보 영역 ROI를 추적해 재검출 안정화

### Marker Confidence Filter와 연계

기체 흔들림이 큰 순간의 인식 결과는 확정 판단에서 낮은 가중치를 준다.

예시:

```text
roll/pitch 변화 작음 -> 인식 결과 높은 가중치
roll/pitch 변화 큼 -> 인식 결과 낮은 가중치
yaw rate 큼 -> ID만 후보 저장, 위치 확정 보류
```

### 문서용 표현

> 짐벌 미탑재로 인해 발생할 수 있는 영상 흔들림은 IMU 기반 프레임 가중치 조정, optical flow 안정화, PX4 제어 파라미터 튜닝 및 필요 시 소스 레벨 제어 로직 개선을 통해 보완한다.

## 12. 경량화 전략

### 핵심 방향

고성능 보드, 짐벌, 대용량 배터리 등 중량 증가 요인을 최소화하고, 가벼운 기체의 빠른 동역학 응답성을 활용한다.

### 줄이거나 제외할 요소

- 2축 또는 3축 짐벌
- 과도하게 무거운 고성능 GPU 보드
- 필요 이상의 대용량 배터리
- 무거운 카메라 마운트
- 불필요한 외장 센서
- 복잡한 보조 구조물

### 우선 검토할 구성

- 경량 쿼드로터 프레임
- 고정식 하향 카메라
- Raspberry Pi급 경량 온보드 컴퓨터 또는 최소 사양 Jetson 계열
- 임무 시간에 맞춘 최적 용량 배터리
- 간단하고 단단한 카메라 마운트
- 짧고 정리된 배선

### 경량화의 장점

- 같은 추력에서 더 높은 가속도 확보
- 짧은 제동 거리 확보
- 방향 전환 응답성 향상
- 배터리 효율 향상
- 스프린트 비행 효과 극대화
- 빠른 자세 안정화 가능

### 단점과 보완

경량 기체는 외란, 진동, 바람에 민감할 수 있다. 또한 짐벌이 없으면 영상 흔들림이 증가할 수 있다. 이를 다음 방식으로 보완한다.

- PX4 제어 튜닝
- Anti-Sway 제어 모드
- IMU 기반 영상 안정화
- Optical Flow 기반 보정
- Marker Confidence Filter
- 필요 시 PX4 소스 레벨 수정

### 문서용 표현

> 본 팀은 고성능 보드, 짐벌, 대용량 배터리 등 중량 증가 요인을 최소화하고, 경량 기체 기반의 빠른 동역학 응답성과 소프트웨어 기반 비전 안정화 기술을 활용하여 임무 수행 시간을 단축하는 것을 목표로 한다.

## 12-1. 상용 기체 기반 최소 추가 하드웨어 전략

### 방향 수정

본 프로젝트의 중심은 기체 자체 제작이 아니라 소프트웨어 기반 자율비행, 인식 안정화, 경로 최적화이다. 따라서 기체는 완전 자체 제작보다 검증된 상용 기체 또는 상용 프레임 기반 기체를 활용하고, 필요한 센서와 온보드 컴퓨터만 최소한으로 추가하는 방향이 현실적이다.

### 기본 방침

- 기체 구조, 모터, ESC, 배터리 시스템은 검증된 상용 구성을 우선 사용
- 팀의 개발 자원은 항법, 인식, 제어, 미션 매니저에 집중
- 추가 장비는 하향 카메라, 경량 온보드 컴퓨터, 전원 모듈, 통신 모듈 정도로 제한
- 짐벌, 고성능 GPU 보드, 대형 배터리, 과도한 외장 센서는 배제
- 기체 개조는 무게중심, 진동, 냉각, 안전성에 영향을 주지 않는 범위에서 수행

### 상용 기체 선정 기준

| 항목 | 요구 사항 | 이유 |
|------|-----------|------|
| 제어 접근성 | MAVLink, PX4 Offboard, ArduPilot Guided, SDK 제어 중 하나 지원 | 자율 이착륙 및 경로 추종 명령 필요 |
| 탑재 여유 | 추가 장비 80~150g 이상 탑재 가능 | 카메라, 보드, 전원 모듈 장착 필요 |
| 하향 장착 공간 | 기체 하부에 카메라 장착 가능 | 격자선과 ArUco 인식 필요 |
| 진동 특성 | 카메라 영상에 심한 젤로 현상이 없어야 함 | 마커 인식률 유지 |
| 실내/저고도 안정성 | 규정 고도 2m에서 안정적인 위치 및 자세 유지 | 인식 안정성과 미션 조건 동시 대응 |
| 수동 조종 백업 | 조종기 override 또는 kill switch 가능 | 테스트 및 비상 상황 대응 |
| 로그 접근성 | IMU, 자세, 속도, setpoint 로그 확인 가능 | 제어 튜닝 및 실패 분석 |
| 배터리 교체성 | 짧은 테스트 반복이 쉬워야 함 | 개발 효율 향상 |

### 상용 기체 후보군

#### A안: PX4/ArduPilot 지원 상용 프레임 또는 RTF 기체

가장 추천되는 방향이다. 완성 기체 또는 반완성 기체를 사용하되, 비행 컨트롤러가 Pixhawk 계열이거나 PX4/ArduPilot을 지원하면 ROS2와 MAVLink 연동이 쉽다.

장점:

- Offboard 또는 Guided 모드로 속도, 위치, 가속도 setpoint 전달 가능
- PX4 파라미터 튜닝 및 로그 분석 가능
- 필요 시 PX4 FlightTask 또는 Position Controller 수정 가능
- 대회용 완전 자율 미션 구현에 적합

주의점:

- DJI Avata 같은 완제품보다 초기 세팅 난이도가 높을 수 있음
- 프레임, 배터리, propeller guard, 카메라 마운트 선정 필요
- 튜닝이 부족하면 흔들림이 커질 수 있음

적합한 형태:

```text
상용 3~5인치 쿼드 프레임 또는 RTF 기체
-> Pixhawk 계열 FC
-> PX4 또는 ArduPilot
-> ROS2 companion computer
-> 하향 고정 카메라
```

#### B안: DJI Avata 계열 같은 소형 FPV 상용기

DJI Avata 계열은 가볍고 빠르며 덕트형 구조로 안전성이 좋다는 장점이 있다. 다만 공식 SDK 호환성 측면에서 Avata 및 Avata 2는 자율비행 제어 접근이 제한적이므로, 대회용 완전 자율비행 플랫폼으로 바로 쓰기에는 리스크가 크다.

활용 가능 방향:

- 기체 동역학, 경량화, 덕트형 구조 참고
- 조종자 보조 실험 또는 데이터 수집용 플랫폼
- 하향 카메라와 온보드 보드를 부착한 인식 알고리즘 테스트
- 최종 대회 기체가 아니라 개념 검증용 테스트베드

한계:

- SDK 미지원 모델은 ROS2에서 직접 속도/위치 setpoint를 넣기 어려움
- PX4 소스 수정 또는 내부 제어기 튜닝 불가
- 자율 이착륙, 격자 추종, 마커 기반 재방문 구현에 제약
- 추가 장비 장착 시 제조사 보증, 안전성, 센서 간섭 문제 발생 가능

정리:

```text
Avata 계열 = 좋은 참고 모델 또는 보조 테스트베드
PX4/ArduPilot 계열 상용기 = 실제 자율 미션 구현에 적합
```

#### C안: DJI SDK 지원 기체

DJI 일부 모델은 Mobile SDK 또는 Enterprise 계열 SDK를 지원한다. 다만 대회 미션은 GPS 비의존 저고도 격자 추종과 빠른 setpoint 제어가 핵심이므로, SDK 지원 여부뿐 아니라 실시간 제어 주기, 실내 비행 안정성, 기체 크기, 추가 하드웨어 장착성까지 함께 검토해야 한다.

장점:

- 완성도 높은 상용 기체 안정성
- 일부 모델은 카메라, 배터리, 로그 관리가 편함

한계:

- SDK가 있어도 PX4 수준의 저수준 제어 수정은 어려움
- Enterprise 계열은 무겁고 가격이 높을 수 있음
- 대회 미션의 경량 스프린트 컨셉과 맞지 않을 수 있음

### 최종 권장 방향

본 프로젝트에는 A안이 가장 적합하다. 즉, 기체 전체를 처음부터 제작하지는 않되, PX4 또는 ArduPilot 접근이 가능한 상용 기반 기체를 사용한다. 이렇게 하면 하드웨어 제작 부담은 줄이면서도, 핵심 소프트웨어인 GPS 비의존 항법, 스프린트 경로 계획, 인식 친화 제어를 충분히 구현할 수 있다.

```text
상용 베이스 기체
-> PX4/ArduPilot 접근 가능
-> 최소 추가 하드웨어 장착
-> 자율비행 소프트웨어 자체 개발
```

### 추가 하드웨어 구성

#### 1. 하향 카메라

목적:

- 격자선 검출
- 격자 교차점 검출
- ArUco 마커 인식
- Optical Flow 기반 이동량 추정
- 버티포트 착륙 보조

권장 조건:

- 글로벌 셔터 카메라 우선 검토
- 최소 720p급 해상도
- 60fps 이상이면 고속 비행 중 인식에 유리
- 렌즈 화각 90~120도 범위 검토
- USB 또는 MIPI CSI 연결
- 하향 고정 장착
- 진동 절연 마운트 적용

비고:

롤링 셔터 카메라도 사용할 수 있지만, 빠른 회전과 진동이 있으면 격자선과 ArUco가 왜곡될 수 있다. 따라서 스프린트 비행을 강조하려면 글로벌 셔터 또는 높은 프레임레이트 카메라가 유리하다.

#### 2. 경량 온보드 컴퓨터

목적:

- OpenCV 기반 ArUco 인식
- 격자선 검출
- Optical Flow 계산
- Marker Confidence Filter
- Mission Manager 실행
- PX4/ArduPilot과 MAVLink 또는 ROS2 통신

후보:

- Raspberry Pi 4 또는 5
- Raspberry Pi Compute Module 계열
- Orange Pi 계열
- Jetson Orin Nano 계열

선정 기준:

| 기준 | 설명 |
|------|------|
| 무게 | 기체 가속 성능을 해치지 않을 것 |
| 전력 | 별도 5V 전원으로 안정 공급 가능할 것 |
| 발열 | 장시간 테스트에서 thermal throttling이 적을 것 |
| 카메라 입력 | USB 또는 CSI 카메라 연결 가능 |
| ROS2 지원 | Linux 및 ROS2 실행 가능 |
| OpenCV 성능 | 목표 프레임레이트에서 ArUco 검출 가능 |

운용 전략:

- 초기 개발은 Raspberry Pi급 보드로 시작
- 프레임레이트가 부족하면 ROI 처리, 프레임 스킵, 해상도 조정 적용
- 그래도 부족하면 Jetson 계열로 확장
- 고성능 보드 사용은 마지막 선택지로 둠

#### 3. 전원 모듈

목적:

- 온보드 컴퓨터와 카메라에 안정적인 전원 공급
- ESC 및 모터 노이즈가 영상/보드에 전달되는 것 방지

권장 구성:

- 메인 배터리에서 5V DC-DC buck converter 사용
- 온보드 컴퓨터용 5V 3A 이상 확보
- Jetson 사용 시 보드 요구 전류에 맞춰 5V 또는 9~20V 입력 검토
- 카메라와 보드 전원은 가능한 한 안정적인 레귤레이터 사용
- 전원선은 짧게 정리하고 신호선과 분리

주의점:

- 전압 강하가 발생하면 보드 리부팅으로 미션 실패 가능
- 모터 급가속 시 전원 노이즈가 커질 수 있음
- 전원 모듈 무게와 발열도 함께 고려해야 함

#### 4. 통신 모듈

목적:

- 실시간 상태 모니터링
- 로그 다운로드
- 비상 정지 또는 모드 전환
- QGroundControl 또는 개발 PC와 연결

후보:

- Wi-Fi
- 915MHz 또는 433MHz telemetry radio
- RC 조종기 수동 override
- USB 연결 기반 지상 테스트

운용 전략:

- 실제 미션 제어는 온보드에서 독립 수행
- 지상국은 모니터링과 비상 개입 위주
- 통신 끊김 시에도 안전 호버 또는 귀환 로직 수행

#### 5. 고도 보조 센서

목적:

- 규정 고도 2m 유지
- 착륙 직전 고도 보정
- Optical Flow scale 보정

후보:

- 기압계
- ToF 거리 센서
- 소형 LiDAR rangefinder
- 하향 초음파 센서

운용 전략:

- 기본은 FC의 기압계와 자세 추정 사용
- 2m 근처 저고도 정밀도가 부족하면 소형 rangefinder 추가
- rangefinder는 2m 부근에서 노이즈와 응답 지연을 사전에 확인
- 고도 센서는 필수 추가 장비가 아니라 성능 보완 옵션으로 둠

#### 6. 카메라 마운트

목적:

- 하향 카메라 고정
- 진동 완화
- 무게중심 영향 최소화
- 렌즈 방향 유지

권장 조건:

- 3D 프린팅 경량 마운트
- 고무 댐퍼 또는 얇은 진동 흡수재 적용
- 기체 중심축 근처에 배치
- 배터리 교체와 충돌하지 않게 설계
- 시야를 프레임, 랜딩기어, 프로펠러가 가리지 않게 배치

### 추가 하드웨어 무게 예산

초기 목표는 추가 장비 전체를 100g 내외로 제한하는 것이다. 기체 탑재 여유에 따라 150g까지 확장할 수 있지만, 스프린트 비행 컨셉을 유지하려면 가능한 한 가볍게 구성한다.

| 구성품 | 목표 무게 |
|--------|-----------|
| 하향 카메라 | 5~20g |
| 카메라 렌즈 및 케이블 | 5~15g |
| 온보드 컴퓨터 | 25~70g |
| 전원 모듈 | 10~25g |
| 통신 모듈 | 5~20g |
| 카메라/보드 마운트 | 10~30g |
| 여유분 | 10~30g |

목표:

```text
1차 목표 추가 중량: 100g 이하
허용 상한 추가 중량: 150g 이하
```

### 장착 위치 원칙

- 추가 장비는 기체 무게중심에 최대한 가깝게 배치
- 하향 카메라는 기체 중심 아래에 배치
- 보드는 프로펠러 다운워시와 모터 열에 직접 노출되지 않게 배치
- 전원선은 짧게, 카메라 신호선은 모터/ESC 전원선과 분리
- 배터리 교체가 쉬운 구조 유지
- 착륙 시 카메라가 지면에 닿지 않도록 보호 구조 확보

### 하드웨어 인터페이스

#### PX4/ArduPilot 기반 기체

```text
Companion Computer
  -> USB/UART MAVLink
  -> Flight Controller
  -> Motor/ESC
```

가능 기능:

- 속도 setpoint 전달
- 위치 setpoint 전달
- yaw setpoint 전달
- 모드 전환
- arm/disarm
- 상태 및 로그 수신
- failsafe 연동

#### DJI Avata 계열

```text
Companion Computer
  -> Camera processing only
  -> Flight control access limited
```

가능 기능:

- 별도 하향 카메라 기반 인식 실험
- 비행 로그와 영상 기반 알고리즘 검증
- 경량 장착 구조 테스트

제한 기능:

- ROS2에서 직접 PX4 Offboard처럼 제어하기 어려움
- 기체 제어기 내부 수정 불가
- 자동 임무 수행 명령 삽입이 어려움

### 개발 단계별 하드웨어 계획

| 단계 | 목표 | 기체/하드웨어 |
|------|------|---------------|
| 1단계 | 영상 인식 알고리즘 검증 | 지상 테스트 리그, 하향 카메라, 노트북 |
| 2단계 | 상용기 탑재 인식 테스트 | 상용 드론 하부에 카메라/보드 임시 장착 |
| 3단계 | 자율제어 연동 | PX4/ArduPilot 접근 가능한 기체 사용 |
| 4단계 | 스프린트 제어 튜닝 | 속도/가속도/감속도 프로파일 적용 |
| 5단계 | 최종 경량화 | 마운트, 배선, 보드, 배터리 무게 최적화 |

### 하드웨어 검증 체크리스트

- [ ] 추가 장비 장착 후 무게중심이 크게 틀어지지 않는가
- [ ] 카메라 시야에 기체 구조물이 들어오지 않는가
- [ ] 급가속 시 온보드 컴퓨터 전원이 꺼지지 않는가
- [ ] 모터 진동으로 영상이 심하게 흔들리지 않는가
- [ ] 카메라 프레임레이트가 목표 속도에서 충분한가
- [ ] 보드 발열로 성능 저하가 발생하지 않는가
- [ ] 수동 조종 override가 항상 가능한가
- [ ] 통신이 끊겨도 안전 모드로 들어가는가
- [ ] 착륙 시 카메라와 보드가 지면에 닿지 않는가

### 문서용 표현

> 본 팀은 기체 자체 제작보다 검증된 상용 기체 또는 상용 프레임 기반 플랫폼을 활용하고, 하향 카메라와 경량 온보드 컴퓨터 등 최소한의 추가 하드웨어만 장착한다. 이를 통해 하드웨어 개발 리스크를 줄이고, 개발 역량을 GPS 비의존 항법, 스프린트 경로 계획, 인식 친화 제어 알고리즘에 집중한다.

### 확인 메모

- DJI 공식 지원 페이지 기준 DJI Avata는 약 410g의 이륙중량과 18분 내외의 hover time을 갖는 소형 FPV 기체로, 경량 스프린트 컨셉의 참고 모델로 적합하다.
- DJI Avata 2 공식 사양 기준 최대 비행 시간은 약 23분이며, 배터리 무게는 약 145g이다.
- DJI 공식 SDK 호환표 기준 DJI Avata와 DJI Avata 2는 SDK Compatibility가 `No`로 표시되어 있으므로, ROS2 기반 완전 자율 제어 플랫폼으로 바로 쓰기에는 제약이 있다.

## 14. 마커 탐색 및 구조 경로

### 탐색 단계

- 격자선을 따라 이동
- 단순 DFS/BFS가 아니라 물리 기반 최소 시간 탐색 사용
- 탐색하지 않은 구간을 커버하되, 실제 예상 비행 시간이 짧은 경로 우선
- 긴 직선 구간은 Sprint Mode로 묶어 가속, 순항, 감속 프로파일 적용
- 회전, 정지, 재가속이 많은 경로는 비용을 높게 반영
- 마커 후보 발견 시 감속 후 인식 안정 모드 진입
- 확정된 마커는 ID와 격자 좌표 저장

### 구조 경로 단계

4개 마커를 모두 찾으면 ID 역순으로 정렬한다.

```text
Detected IDs: 3, 8, 1, 5
Visit Order: 8 -> 5 -> 3 -> 1
```

구조 경로에서는 격자선을 반드시 따라갈 필요가 없으므로, 저장된 마커 좌표 사이를 빠르게 이동할 수 있다. 이때도 가속도 기반 속도 프로파일을 적용한다.

구조 경로 역시 단순 직선거리 최단 경로만 사용하지 않는다. 목표 마커 간 이동에 대해 현재 속도, 제동 거리, 회전각, 접근 안정화 시간을 반영하여 실제 도착 시간이 가장 짧은 경로를 선택한다.

### 구조 경로에서의 안정화

- 목표 마커 접근 전 감속
- 저장 좌표 근처에서 하향 카메라 재확인
- Marker Confidence Filter로 재방문 확인
- 각 마커 위치에서 3초 호버링

## 17. 버티포트 착륙 보조 아이디어

### 기본 방향

GPS 없이 복귀 및 착륙해야 하므로, 버티포트를 단순히 좌표로 저장하는 것이 아니라 `Home Pose`와 `시각 재획득 대상`으로 함께 관리한다. 드론은 이륙 직후 버티포트 중심을 기준으로 로컬 좌표계를 초기화하고, 미션 중에는 격자선과 ArUco 마커로 누적 오차를 보정한다. 귀환 단계에서는 저장된 Home Pose로 먼저 접근한 뒤, 최종 구간에서 하향 카메라로 버티포트를 다시 검출해 중심 오차를 보정한다.

### 핵심 아이디어

```text
이륙 직후 Home Pose 초기화
-> 격자 좌표계와 Home Pose 사이의 변환 저장
-> 미션 중 Grid Localization으로 누적 오차 보정
-> 귀환 시 Home Pose 근처까지 예측 이동
-> 하향 카메라로 버티포트 재획득
-> 중심 오차 기반 비전 서보 착륙
```

### 1단계: Home Pose 초기화

드론은 버티포트 위에서 이륙하므로, 이륙 직후 2m 고도에서 짧게 호버링하며 현재 위치를 `Home Pose`로 저장한다.

저장 항목:

- 이륙 지점 기준 로컬 위치
- 드론 yaw 방향
- 하향 카메라 기준 버티포트 중심 픽셀 위치
- 버티포트 원형 외곽선 또는 착륙 마커 관측 정보
- IMU/Optical Flow 기준 초기 위치 추정값

가능하면 이륙 직후 하향 카메라로 버티포트 원형 외곽선을 검출해, 드론이 실제로 버티포트 중심 위에 있는지 확인한다. 중심에서 벗어난 상태로 이륙했다면 초기 Home Pose에 그 오차를 함께 저장한다.

```text
HomePose = {
  position_in_local_frame,
  yaw_reference,
  vertiport_center_offset,
  confidence
}
```

### 2단계: 격자 좌표계와 Home Pose 연결

버티포트는 미션 구역 외부에 있으므로, 미션 구역의 격자 좌표계와 버티포트 위치 사이의 상대 변환을 알아야 한다. 이를 위해 이륙 후 미션 구역으로 진입하는 동안 Visual-Inertial Odometry와 Optical Flow를 이용해 이동량을 추정하고, 처음 격자선 또는 격자 교차점을 인식하는 순간 격자 좌표계와 Home Pose 사이의 변환을 확정한다.

```text
T_home_grid = Home 좌표계 -> Grid 좌표계 변환
T_grid_home = Grid 좌표계 -> Home 좌표계 변환
```

이후 드론의 위치는 다음 두 기준으로 함께 관리한다.

- `Home 기준 위치`: 버티포트까지 돌아가기 위한 기준
- `Grid 기준 위치`: 격자 탐색과 마커 위치 저장을 위한 기준

### 3단계: 미션 중 누적 오차 보정

GPS가 없으면 IMU 적분만으로는 위치 drift가 커질 수 있다. 따라서 미션 중에는 다음 관측값으로 위치 추정을 계속 보정한다.

- 격자선 방향
- 격자 교차점 위치
- ArUco 마커 위치
- Optical Flow 이동량
- 2m 고도 기준 scale
- yaw 변화량

격자 교차점이나 ArUco 마커처럼 위치가 명확한 기준점을 관측할 때마다 현재 위치 추정값을 보정한다. 이 보정 결과는 `Grid 기준 위치`뿐 아니라 `Home 기준 위치`에도 반영된다.

### 4단계: 예측 귀환 경로 생성

마커 순회를 마치면 현재 `Grid 기준 위치`에서 저장된 Home Pose로 돌아가는 경로를 생성한다. 이때도 단순 직선거리 최단 경로가 아니라 Minimum-Time Sprint Search Planner를 사용한다.

귀환 경로 비용:

```text
ReturnCost =
  T_sprint_to_home
  + T_turn
  + T_deceleration
  + T_home_reacquire
  + T_landing
```

귀환 중에는 Home Pose에 가까워질수록 속도 제한을 강화한다.

```text
Home approach distance > 5m: Sprint return
Home approach distance <= 5m: 감속
Home approach distance <= 3m: 버티포트 탐색 모드
Home detected: 비전 서보 착륙
```

### 5단계: 버티포트 재획득

저장된 Home Pose 근처에 도착하면 하향 카메라로 버티포트를 다시 찾는다. 버티포트는 지름 3m의 원형 구조이므로, 2m 고도에서 넓은 화각 카메라를 사용하면 원형 외곽선 또는 중심 영역을 검출할 수 있다.

검출 방법:

- 원형 외곽선 검출
- 색상 또는 명암 대비 기반 segmentation
- Hough circle 또는 ellipse fitting
- 별도 착륙 마커 사용 가능 시 ArUco/AprilTag 검출
- Optical Flow 기반 정지 상태 확인

버티포트를 바로 찾지 못하면 예측 Home Pose 주변에서 작은 탐색 패턴을 수행한다.

```text
Predicted Home 도착
-> 원형 외곽선 미검출
-> 속도 제한
-> 작은 사각형 또는 나선형 재탐색
-> 버티포트 재획득
```

### 6단계: 중심 오차 기반 비전 서보 착륙

버티포트가 검출되면 이미지 중심과 버티포트 중심의 픽셀 오차를 계산하고, 이를 수평 속도 명령으로 변환한다.

```text
error_x = image_center_x - vertiport_center_x
error_y = image_center_y - vertiport_center_y

vx_cmd = kx * error_x
vy_cmd = ky * error_y
```

착륙 조건:

- 버티포트 중심 오차가 기준 이하
- 수평 속도가 기준 이하
- yaw rate가 기준 이하
- roll/pitch 흔들림이 기준 이하
- 버티포트 검출 confidence가 기준 이상

조건을 만족하면 단계적으로 하강한다. 조건이 깨지면 하강을 멈추고 다시 중심 정렬을 수행한다.

### 7단계: 실패 대응

| 상황 | 대응 |
|------|------|
| Home Pose 예측 오차 증가 | 마지막 격자 교차점 기준으로 위치 재보정 후 재귀환 |
| 버티포트 미검출 | 예측 위치 주변 작은 사각형/나선형 탐색 |
| 버티포트 중심 검출 불안정 | 2m 고도에서 호버링 후 여러 프레임 누적 |
| 착륙 중 중심 오차 증가 | 하강 중지, 수평 보정 후 재하강 |
| 배터리 부족 | 버티포트 예측 위치로 저속 접근 후 안전 착륙 |

### 카메라 화각 요구

2m 고도에서 지름 3m 버티포트를 안정적으로 검출하려면 하향 카메라가 충분한 지면 폭을 볼 수 있어야 한다. 따라서 카메라 선정 시 2m 고도에서 최소 3m 이상의 지면 폭이 영상에 들어오는지 확인한다.

```text
권장 조건:
2m 고도에서 관측 가능한 지면 폭 >= 3m + 위치 오차 여유
```

화각이 부족하면 Home Pose 근처에서 고정 호버링 후 작은 탐색 패턴으로 버티포트 중심을 재획득한다.

### 가능 아이디어

- 버티포트 원형 외곽선 검출
- 별도 착륙 마커 사용 가능 여부 검토
- 하향 카메라로 중심점 오차 추정
- 하강 중 수평 오차 보정
- 착륙 직전 속도 제한 및 자세 안정화

### 문서용 표현

> GPS 기반 복귀 대신 이륙 직후 저장한 Home Pose와 미션 중 보정되는 Grid Localization을 결합해 버티포트 근처까지 귀환한다. 최종 접근 단계에서는 저장 좌표만 신뢰하지 않고, 하향 카메라로 지름 3m 버티포트의 원형 외곽선 또는 착륙 마커를 재검출한다. 이후 영상 중심과 버티포트 중심의 픽셀 오차를 수평 속도 명령으로 변환하는 비전 서보 제어를 수행하고, 중심 오차와 흔들림이 기준 이하일 때만 단계적으로 하강한다.

| 짐벌 미탑재로 영상 흔들림 발생 | IMU/Optical Flow 기반 안정화 및 프레임 가중치 조정 |
| 경량 기체가 외란에 민감 | PX4 튜닝, 속도 제한, jerk 제한 적용 |
| PX4 소스 수정 난이도 높음 | 1차는 ROS2 Offboard 제어로 구현, 필요 시 단계적 수정 |
| 온보드 컴퓨팅 성능 부족 | ROI 기반 인식, 프레임 스킵, 경량 알고리즘 적용 |
| 배터리 용량 부족 | 임무 시간 기반 최적 용량 선정 및 긴급 귀환 로직 적용 |
| 상용 FPV 기체의 SDK 제어 제한 | PX4/ArduPilot 접근 가능한 상용 플랫폼 우선 선정 |
| 추가 장비 장착으로 무게중심 변화 | 중심축 근처 장착, 테스트 호버링 후 마운트 조정 |
| 전원 노이즈 또는 전압 강하 | 별도 DC-DC 전원 모듈, 짧은 배선, 전원 로그 확인 |
| 2m 저고도 고속 비행 시 motion blur 증가 | 2m 기준 속도 제한, 높은 FPS 카메라, 조기 감속 적용 |
| 규정 고도 2m 유지 오차 발생 | rangefinder 보조, 고도 제어 튜닝, 속도 제한 적용 |
| GPS 없이 버티포트 복귀 오차 발생 | Home Pose 저장, Grid-Home 변환 유지, 최종 버티포트 재검출 적용 |

## 21. 제출 문서용 통합 문단

본 팀은 화성 탐사 헬리콥터 Ingenuity의 GPS 비의존 항법 철학을 참고하여, GPS에 의존하지 않고 하향 카메라와 IMU를 기반으로 격자 환경을 인식하는 경량 자율비행 시스템을 개발한다. 기체는 완전 자체 제작보다 검증된 상용 기체 또는 상용 프레임 기반 플랫폼을 활용하고, 하향 고정 카메라와 경량 온보드 컴퓨터 등 최소한의 추가 하드웨어만 장착한다. 이를 통해 하드웨어 제작 리스크를 줄이고, 개발 역량을 GPS 비의존 항법, 스프린트 경로 계획, 인식 친화 제어 알고리즘에 집중한다. 직선 격자 구간에서는 가속도 기반 Sprint Navigation을 적용해 임무 시간을 단축하고, 마커 후보가 발견되면 자동 감속 및 Anti-Sway Stabilization Mode로 전환해 안정적인 ArUco 인식을 수행한다. 짐벌 미탑재로 인한 영상 흔들림 문제는 PX4 제어 튜닝 및 필요 시 소스코드 수정, IMU/Optical Flow 기반 영상 안정화, 누적 관측 기반 Marker Confidence Filter를 통해 보완한다. 이를 통해 빠른 이동 성능과 높은 인식 정확도를 동시에 확보하는 대회 특화형 자율비행 시스템을 구현하고자 한다.
